## 一、选图入口总览（共 3 个入口函数，覆盖 6 大场景）

ASA 系统实际只有 **3 个选图入口函数**，通过不同参数覆盖所有场景。以下按入口函数分类：

### 入口 A：`openIconPicker_ic(bi)` — 板块级

| # | 场景 | 调用方式 | 存储位置 |
|---|------|----------|----------|
| A1 | info_card 标题图标 | `openIconPicker_ic(bi)` → `selectInfoCardIcon` | `content_blocks[bi].icon_url` |
| A2 | 板块标题组合图 | A1 选择器中「组合」→「合并」→ `saveComboIC` | 同上（生成 `data:` URL） |

### 入口 B：`openIconPicker_desc(el, bi, di)` — 描述行级

| # | 场景 | 调用方式 | 存储位置 |
|---|------|----------|----------|
| B1 | 描述行图片 | `openIconPicker_desc(el, bi, di)` | `descriptions[di].image_url` |
| B2 | 描述行组合图 | B1 选择器中「组合」→「合并」 | 同上（生成 `data:` URL） |
| B3 | 图标组行内图标 | `openIconPicker_desc(this, bi, ii)` + overlay._igRowDi | `descriptions[rowDi].icons[ii].image_url` |
| B4 | 图标组组合图 | B3 选择器中「组合」→「合并」 | 同上 |

### 入口 C：`openIconPicker(si, bi, ri, ci, ii)` — 部落基地行级

| # | 场景 | 调用方式 | 存储位置 | 编辑器 |
|---|------|----------|----------|--------|
| C1 | 部落基地存储行图标 | `openIconPicker(si, bi, ri)` → `selectDeviceIcon` | `rows[ri].device_icon_url` | 部落基地 |
| C2 | 部落基地物品图标 | `openIconPicker(si, bi, ri, ci, ii)` → `selectItemIcon` | `rows[ri].categories[ci].items[ii].icon_url` | 部落基地 |
| C3 | 部落基地组合图 | C1/C2 选择器中「组合」→ `saveComboDevice` | 同上（生成 `data:` URL） | 部落基地 |

> ⚠️ **入口 C 不涉及反色逻辑**，仅做 URL 写入。`saveComboDevice` 也无反色。

### `composeIcons` 分发总表

| onclick 匹配 | 分发函数 | 对应场景 |
|-------------|----------|----------|
| `selectDescImage` + igRowDi | `saveComboIG` | B4 图标组组合图 |
| `selectDescImage` | `saveComboDesc` | B2 描述行组合图 |
| `selectInfoCardIcon` | `saveComboIC` | A2 板块标题组合图 |
| `selectDeviceIconFromEl` | `saveComboDevice` | C3 部落基地组合图 |

### 非选择器方式：supply_card / card_grid / server_grid

以下三种板块类型的图标**不使用图标选择器**，而是纯文本 `<input>` 手动输入 URL：

| 板块类型 | 图标字段 | 输入方式 |
|----------|----------|----------|
| `supply_card` | `item.icon_url` / `locations[].image_url` | `<input id="mcSCIcon{bi}">` / `<input id="mcSCLocImg{bi}_{sid}_{li}">` |
| `card_grid` | `cards[ci].image_url` | `<input id="mcCGImg{bi}_{ci}">` |
| `server_grid` | `items[ii].icon_url` | `<input id="mcSGItemIco{bi}_{ii}">` |

> 这三种板块类型在 `MC_TYPE_LABELS` 中已被移除，不再能通过"+ 板块"按钮新增。仅保留编辑和渲染兼容。

## 三、三态渲染：默认态 / 线性态 / 色块态

lovelace 预览区有三种渲染模式，影响图片的反色 CSS 类名前缀：

| 模式 | block_maps | 图片 CSS class |
|------|:---:|------|
| **默认态** | `None` | `ic-desc-img` — 包裹在 `<span class="ic-desc-wrap ic-mode-xxx">` |
| **线性态** | `st=1` | `ic-block-img` — 直接渲染，模式类在 img 上 |
| **色块态** | `st=2` | `ic-block-img` — 直接渲染，模式类在 img 上 |

### 渲染差异

HTML 生成逻辑由 `build_lovelace.py` 中的 `render_tab_html()` 统一处理。`preview_server.py` 通过 `from build_lovelace import render_tab_html` 导入，**HTML 结构完全一致**。

但 CSS 在**两个文件中各自独立定义**：

| 文件 | CSS 位置 | 选择器格式 |
|------|----------|-----------|
| `build_lovelace.py` | 全局 CSS 块 + `IC_CSS` | `img.ic-auto-dark` + `ha-card .info-card-block img.ic-auto-dark` |
| `preview_server.py` | 独立 CSS 块 | 仅 `ha-card .info-card-block img.ic-auto-dark` |

> ⚠️ **CSS 双文件重复**：修改反色 CSS 时必须两处同步，详见 [CSS 调试铁律](../.copilot/instructions/general.instructions.md)。

```python
# build_lovelace.py（两处都生成）
if block_maps:
    # 线性态/色块态：ic-block-img，模式类直接在 img 上
    _img_tag = '<img src="{}" class="ic-block-img{}{}" />'.format(url, auto_cls, mode_cls)
else:
    # 默认态：ic-desc-img，包裹 SPAN wrapper
    _img_tag = '<span class="ic-desc-wrap{}"><img src="{}" class="ic-desc-img{}" /></span>'.format(mode_cls, url, auto_cls)
```

### CSS 选择器差异（双文件同步）

| 模式 | build_lovelace.py | preview_server.py |
|------|-------------------|-------------------|
| 默认态反色 | `img.ic-auto-dark` | `ha-card .info-card-block img.ic-auto-dark` |
| 色块态反色 | `ha-card .info-card-block img.ic-auto-dark` | 同上 |

### 色块态额外效果

色块态下的 `ic-qty` 数量徽章、`ic-badge-hollow` 空心序号、标题分隔线 `ig-title-line` 等均有独立样式。

## 四、正反色 vs 逆反色

| 模式 | `image_auto_color_mode` | CSS mode_cls | 含义 |
|------|:---:|------|------|
| **关闭** | `off` | (无) | 不进行任何反色 |
| **正反色** | `normal` | `ic-mode-normal` | 跟随主题：暗主题下深色图反白 |
| **逆反色** | `reverse` | `ic-mode-reverse` | 反向：暗主题下浅色图反白 |

### 自动转换规则

在 `saveMCBlocks` 中，**色块态**下的 `normal` 会被自动转为 `reverse`：

```javascript
if (mode === 'normal') {
    var descSrv = document.getElementById('mcICDescServer'+bi+'_'+origDi)?.value || '';
    if (descSrv) mode = 'reverse';
}
```

> 原因：色块态卡片有彩色背景，正反色白字在彩色底上不可见，逆反色能保证对比度。

### 视觉效果

亮暗主题均有处理，通过 `ic-auto-dark` / `ic-auto-light` 两个独立类区分：

| 模式 | CSS 类 | 暗主题 `[data-theme="dark"]` | 亮主题 `[data-theme="light"]` |
|------|--------|------|------|
| 关闭 | (无) | 原图 | 原图 |
| 正反色·暗图 | `ic-mode-normal` + `ic-auto-dark` | `invert(1)` 反白 | 原图 |
| 正反色·亮图 | `ic-mode-normal` + `ic-auto-light` | 原图 | `invert(1)` 反色 |
| 逆反色·暗图 | `ic-mode-reverse` + `ic-auto-light` | 原图 | `invert(1)` 反色 |
| 逆反色·亮图 | `ic-mode-reverse` + `ic-auto-dark` | `invert(1)` 反白 | 原图 |
| 推定·正反色 | `ic-mode-normal` + `ic-auto-color` | `var(--ic-icon-filter)` | `var(--ic-icon-filter)` |
| 推定·逆反色 | `ic-mode-reverse` + `ic-auto-dark` | `invert(1)` | 原图 |

核心 CSS：
```css
img.ic-auto-dark  { filter: none; }
img.ic-auto-light { filter: none; }
[data-theme="dark"]  img.ic-auto-dark  { filter: invert(1); }
[data-theme="light"] img.ic-auto-light { filter: invert(1); }
```

### 编辑器角标颜色含义

CSS 通过 `::after` 伪元素在图片右上角显示小圆点：

| 类 | 颜色 | 含义 |
|----|------|------|
| `.ic-mode-normal::after` | `var(--accent)` 蓝色 | 正反色已开启，亮度检测成功 |
| `.ic-mode-reverse::after` | `var(--warning)` 橙色 | **逆反色**已开启——反转了正反色方向，常用于色块态保证彩色底对比度 |
| `.ic-mode-failed::after` | `#ef4444` 红色 | 亮度检测失败 |

## 五、两种反色逻辑分支

### 命名

| 分支名 | 条件 | 决策依据 |
|--------|------|----------|
| **「实测反色」** | `lum != None` 且 `lum != 0.5` | 基于实际检测的亮度值 |
| **「推定反色」** | `lum == None` 或 `lum == 0.5` | 仅凭 reverse 标志推断 |

### 实测反色（Measured Inversion）

```python
# build_lovelace.py 行 2356-2358
if _dac_lum is not None and _dac_lum != 0.5:
    _use_lum = (1 - _dac_lum) if _is_rev else _dac_lum
    _dac_cls = ' ic-auto-light' if _use_lum > 0.5 else ' ic-auto-dark'
```

**逻辑**：
1. 取实际亮度值 `_dac_lum`（0~1，由 `detectImageBrightness` 在前端测得）
2. 如果是逆反色 `reverse`，亮度取反 `1 - lum`
3. 有效亮度 > 0.5 → `ic-auto-light`（图偏亮，亮主题下反色）
4. 有效亮度 ≤ 0.5 → `ic-auto-dark`（图偏暗，暗主题下反色）

**适用场景**：图片 URL 为远程 HTTP 地址时，浏览器能加载并检测亮度。

---

### 推定反色（Default Inversion）🗑️ 待清理

> ⚠️ **计划清理**：推定反色分支中的 `ic-auto-color` 类使用 CSS 变量 `--ic-icon-filter`，不产生可见的 `invert()` 效果，调试时无法直观判断反色是否生效。计划后续改为统一走实测反色分支。

```python
# build_lovelace.py 行 2360
else:
    _dac_cls = ' ic-auto-dark' if _is_rev else ' ic-auto-color'
```

**逻辑**：
1. 无亮度数据时，无法判断图片明暗
2. 逆反色 → `ic-auto-dark`（暗主题下反色）
3. 正反色 → `ic-auto-color`（使用 HA 变量 `--ic-icon-filter`，通常跟随主题文字色）

**适用场景**：
- 刚设置反色但亮度检测未完成
- 图片为 `data:image/png;base64`（组合图），亮度检测不可用或未运行
- `image_native_luminance` 未写入 editorData（如组合图非反色模式）

---

### 两种分支对比

| 维度 | 实测反色 | 推定反色 |
|------|----------|----------|
| 条件 | 有亮度值且≠0.5 | 无亮度值或=0.5 |
| 正反色结果 | 暗图→`ic-auto-dark`；亮图→`ic-auto-light` | `ic-auto-color` |
| 逆反色结果 | 暗图→`ic-auto-light`；亮图→`ic-auto-dark` | `ic-auto-dark` |
| CSS 含义 | 仅在对应主题下 `invert(1)` | 跟随主题文字色或固定反色 |
| 依赖 | 前端 `detectImageBrightness` | 无需额外检测 |

### 亮度检测流程

```
用户切换「反色：开」
  → toggleIconColorModeDesc
    → _persistDescMode('normal')  // 同步写入 editorData
    → detectImageBrightness(url, callback)
      → 加载图片到离屏 Canvas
      → 逐像素采样（排除透明像素）
      → 计算平均亮度 (0~1)
      → 回调写入 image_native_luminance
      → 应用 ic-auto-dark / ic-auto-light 视觉类
      → toast "亮度检测成功 [0.XXX]"
```

## 八、改造计划

### 8.1 统一 `openIconPicker` 入口函数

**目标**：将现有 3 个入口函数合并为 1 个，统一参数签名和选择器 HTML 模板。

#### 统一参数签名

```javascript
// 新统一入口
function openIconPicker(ctx) {
    // ctx 为上下文对象，覆盖所有场景
    // ctx.mode: 'info_card_title' | 'desc_row' | 'icon_group' | 'base_row' | 'base_item'
    // ctx.bi: 板块索引（所有场景必传）
    // ctx.di: 描述行索引（desc_row / icon_group 场景）
    // ctx.si, ctx.ri, ctx.ci, ctx.ii: 部落基地多级索引（base 场景）
}
```

#### 统一选择器 HTML

当前问题：三个函数各自拼接 HTML 字符串，维护时需改三处。

统一方案：抽取**单选图器模板函数** `renderIconPickerHTML(ctx)`，所有入口共用一套 HTML：

```javascript
function renderIconPickerHTML(ctx) {
    var html = '<div class="icon-picker-panel">';
    html += renderPickerHeader(ctx);   // 搜索框 + 按钮行
    html += renderPickerCats(cats);    // 分类标签
    html += renderPickerGrid(list, ctx); // 图标网格（onclick 根据 ctx.mode 分发）
    html += '</div>';
    return html;
}
```

按钮行根据 `ctx.mode` 动态显示：

| 按钮 | info_card / desc / icon_group | base_quick_ref |
|------|:--:|:--:|
| 搜索 | ✅ | ✅ |
| 组合 | ✅ | ✅ |
| 刷新 | ✅ | ✅ |
| 反色 | ✅ | ❌ |
| 清空 | ✅ | ✅ |

#### 统一分发

图标网格 item 的 onclick 统一为 `selectIcon(item, ctx)`，内部按 `ctx.mode` 分发：

| ctx.mode | 分发到 | 写 editorData |
|----------|--------|--------------|
| `info_card_title` | `writeIcon_url` → `content_blocks[bi]` | `icon_url` + `icon_auto_color_mode` |
| `desc_row` | `writeDescImage` → `descriptions[di]` | `image_url` + `image_auto_color_mode` + `image_native_luminance` |
| `icon_group` | `writeIGIcon` → `descriptions[rowDi].icons[ii]` | 同上 |
| `base_row` | `writeBaseRowIcon` → `rows[ri]` | `device_icon_url` |
| `base_item` | `writeBaseItemIcon` → `rows[ri].categories[ci].items[ii]` | `icon_url` |

#### 影响范围

- 删除 `openIconPicker_ic`、`openIconPicker_desc`、原 `openIconPicker(si,bi,ri,...)`
- 新增 `openIconPicker(ctx)` + `renderIconPickerHTML(ctx)` + `selectIcon(item, ctx)`
- 所有 onclick 调用点统一改为 `openIconPicker({mode:'xxx', bi:N, di:N, ...})`
- `composeIcons` 分发改为按 `ctx.mode` 匹配
- HTML 模板维护从 3 处减少到 1 处

### 8.2 描述行与图标组存储统一评估

当前两种图片存储位置：

| 类型 | 存储路径 | 反色字段 | 示例 |
|------|----------|----------|------|
| 描述行图片 | `descriptions[di].image_url` | `descriptions[di].image_auto_color_mode` | 单图 |
| 图标组图片 | `descriptions[rowDi].icons[ii].image_url` | `descriptions[rowDi].icons[ii].image_auto_color_mode` | 数组 |

**统一可行性**：🟡 中优先级

不同点：
- 描述行是**单图**，图标组是**多图数组**
- 图标组多了 `quantity` 字段，描述行也有 `quantity`（在行级）
- 反色字段名相同（`image_auto_color_mode` / `image_native_luminance`），但路径深度不同

可统一方向：
```
// 描述行改为内嵌单图数组（与图标组对齐）
descriptions[di].images = [{ image_url, quantity, image_auto_color_mode, image_native_luminance }]

// 图标组保持不变
descriptions[rowDi].icons = [{ image_url, quantity, image_auto_color_mode, image_native_luminance }]
```

**影响范围**：
- `build_lovelace.py`：描述行渲染改为遍历 `images[]`，图标组不变
- `saveMCBlocks`：描述行读取路径调整
- `selectDescImage`：写 `images[0]` 替代直接写 `descriptions[di]`
- `saveComboDesc`：同上
- 所有反色 CSS 不变（类名不依赖存储路径）

**收益**：消除描述行/图标组反色处理的三套并行代码，统一为数组遍历。但改动面大，建议放在 supply_card/card_grid/server_grid 清理之后作为第二步。
